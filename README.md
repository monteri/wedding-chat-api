# Wedding Chat Backend

Production-ready Java 21 / Spring Boot 3 backend for real-time wedding chat.

## Tech stack

- Java 21
- Spring Boot 3.x
- Gradle Kotlin DSL
- PostgreSQL 18 (local via Docker Compose)
- Flyway migrations
- Spring Web + Bean Validation
- Spring WebSocket with STOMP
- Spring Data JPA

## Features

- Join wedding by `slug`, `accessCode`, and guest `displayName`
- Guest session token generated by backend
- Message persistence in PostgreSQL
- Message reaction persistence in PostgreSQL
- Floating reactions broadcast live (no long-term persistence)
- Simple admin message delete protected by `ADMIN_PASSWORD`
- Basic in-memory rate limiting for:
  - chat messages
  - floating reactions

## Environment variables

Copy `.env.example` to `.env` for local development:

```bash
cp .env.example .env
```

Spring Boot loads `.env` automatically on startup (`spring.config.import`). `.env` is gitignored; commit only `.env.example`.

| Variable | Required | Default | Purpose |
| --- | --- | --- | --- |
| `PORT` | No | `8080` | HTTP server port |
| `DB_URL` | No | (see `DATABASE_URL`) | PostgreSQL URL (`jdbc:postgresql://…` or `postgresql://…`) |
| `DATABASE_URL` | No | local default in `application.yml` | Railway/Heroku-style URL; used when `DB_URL` is unset |
| `DB_USERNAME` | No | `wedding_chat` | PostgreSQL user |
| `DB_PASSWORD` | No | `wedding_chat` | PostgreSQL password |
| `FRONTEND_ORIGIN` | No | `http://localhost:5173` | Comma-separated CORS + WebSocket allowed origins |
| `ADMIN_PASSWORD` | No | `change-me` | Password for admin delete endpoint |
| `MESSAGE_RATE_LIMIT_PER_MINUTE` | No | `12` | Per-guest message rate limit |
| `FLOATING_REACTION_RATE_LIMIT_PER_MINUTE` | No | `30` | Per-guest floating reaction rate limit |

### Multiple frontend origins

Set `FRONTEND_ORIGIN` as a comma-separated list (no spaces required):

```env
FRONTEND_ORIGIN=http://localhost:5173,http://localhost:3000,https://wedding.example.com
```

## API endpoints

- `GET /api/health`
- `POST /api/admin/weddings` (create wedding + slug + access code)
- `POST /api/weddings/{slug}/join`
- `GET /api/weddings/{slug}/messages`
- `GET /api/admin/weddings/{slug}/messages`
- `DELETE /api/admin/messages/{messageId}`

### Create wedding (admin)

`POST /api/admin/weddings` with header:

- `X-Admin-Password: <ADMIN_PASSWORD>`

```json
{
  "slug": "sara-and-mike",
  "name": "Sara and Mike Wedding",
  "accessCode": "my-secret-code"
}
```

Response:

```json
{
  "id": "uuid",
  "slug": "sara-and-mike",
  "name": "Sara and Mike Wedding",
  "createdAt": "2026-05-21T12:00:00Z"
}
```

Guests join with the **slug** in the URL and the **access code** in the join body (see below). Share the slug and access code with guests (e.g. link `/{slug}` + code at the door).

### Join request example

`POST /api/weddings/sara-and-mike/join`

```json
{
  "accessCode": "my-secret-code",
  "displayName": "Nina"
}
```

Response:

```json
{
  "weddingId": "uuid",
  "guestId": "uuid",
  "displayName": "Nina",
  "sessionToken": "random-token"
}
```

### Get messages

`GET /api/weddings/{slug}/messages` with header:

- `X-Guest-Token: <sessionToken>`

### Admin list messages

`GET /api/admin/weddings/{slug}/messages` with header:

- `X-Admin-Password: <ADMIN_PASSWORD>`

### Admin delete

`DELETE /api/admin/messages/{messageId}` with header:

- `X-Admin-Password: <ADMIN_PASSWORD>`

## Database setup

Flyway migration is located at:

- `src/main/resources/db/migration/V1__init_schema.sql`

Create a wedding via `POST /api/admin/weddings` (see API section), or insert a row manually before joining:

```sql
INSERT INTO weddings (id, slug, name, access_code, created_at)
VALUES (
  gen_random_uuid(),
  'sara-and-mike',
  'Sara and Mike Wedding',
  'my-secret-code',
  now()
);
```

## Java version note

- The app compiles and runs on **Java 21** (Gradle toolchain).
- If your machine only has **Java 25** installed, use **Gradle 9.4+** (included in this repo). Older Gradle 8.x fails to parse JDK 25 when loading `build.gradle.kts`.
- On first build, Gradle may auto-download JDK 21 via the Foojay toolchain resolver.

## Local development

1. Create your env file:

   ```bash
   cp .env.example .env
   ```

2. Start PostgreSQL 18 (Docker):

   ```bash
   docker compose up -d
   ```

   Local Compose uses `postgres:18-alpine` on host port **5433** (mapped to container `5432`). Data is stored under `/var/lib/postgresql` (required for Postgres 18+ images).

   If the container loops with *"PostgreSQL data in /var/lib/postgresql/data (unused mount/volume)"*, you have an old volume from Postgres 16/17. Reset it (dev only — deletes all local DB data):

   ```bash
   docker compose down -v
   docker compose up -d
   ```

   Then re-run Flyway migrations and recreate weddings (e.g. `POST /api/admin/weddings`).

3. Run the app:

   ```bash
   ./gradlew bootRun
   ```

4. Verify:

   ```bash
   curl http://localhost:8080/api/health
   ```

## Docker build and run

Build image:

```bash
docker build -t wedding-chat-backend .
```

Run container:

```bash
docker run --rm -p 8080:8080 \
  -e DB_URL=jdbc:postgresql://host.docker.internal:5432/wedding_chat \
  -e DB_USERNAME=wedding_chat \
  -e DB_PASSWORD=wedding_chat \
  -e FRONTEND_ORIGIN=https://your-frontend.example.com,https://www.your-frontend.example.com \
  -e ADMIN_PASSWORD=replace-me \
  wedding-chat-backend
```

## Railway deployment

1. Push repository to GitHub.
2. Create a new Railway project from the GitHub repo.
3. Add a PostgreSQL service in Railway.
4. In app service variables, set the database connection (either form works):
   - **Option A (recommended):** Add variable `DATABASE_URL` = `${{Postgres.DATABASE_URL}}` (Railway variable reference). Do **not** set `DB_URL` to an empty value.
   - **Option B:** Set `DB_URL` to a JDBC URL, e.g. `jdbc:postgresql://HOST:PORT/railway?sslmode=require`

   Use the full URL from Railway (with `postgresql://` and `//`). A value like `postgresql:postgres:pass@host:5432/railway` without `//` is fixed automatically, but copying the exact **Connect** URL from Railway is safest.
   - `DB_USERNAME` / `DB_PASSWORD` — only for local dev or JDBC URLs **without** embedded credentials. If `DB_URL` / `DATABASE_URL` already contains `postgres:password@…`, **do not** set `DB_USERNAME` to the database name (`railway`); Railway often injects misleading values.

   Railway provides `postgresql://…` URLs; the app converts them to `jdbc:postgresql://…` when creating the datasource. On startup you should see a log line like `Connecting to PostgreSQL at postgres.railway.internal:5432/railway` — if it shows `localhost:5433`, `DB_URL` is not reaching the container (wrong service, empty variable, or old deploy).

   Redeploy after pulling the latest backend image.
   - `FRONTEND_ORIGIN`
   - `ADMIN_PASSWORD`
   - optional rate limit values
5. Railway provides `PORT`; app reads it automatically.
6. Deploy. Flyway runs schema migration on startup.

## Notes

- Rate limiting is intentionally simple and in-memory. For multi-instance production setups, replace with a shared store (for example Redis).
- Floating reactions are real-time only and intentionally not persisted.
